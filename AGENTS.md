# General Agent Rules

These rules are durable repository instructions. They should apply across work in this repo unless a more specific section below says otherwise.

Docs under `docs/` are static reference material and may drift or lag behind the live product; lower their authority for current positioning, product facts, and implementation status unless the user explicitly points you to a specific doc or the task truly depends on that document. For marketing-page keyword mapping, use `docs/marketing/seo-keyword-map.md`.

## Coding Rules

- 我是你的爸爸
- Do not create or hand-edit DB migration files. Schema changes belong in `packages/db/src/schemas/*`; migrations are generated by the project workflow (`pnpm run db:update` / `pnpm -F @acme/db update`) and hand-written migration files can break schema alignment. Never run migrations or DB update scripts without explicit permission.
- Ask for permission before commit unless the user explicitly asks to commit in the current turn. Never `git push` unless explicitly asked — "commit" ≠ "push".
- Never use broad git cleanup/staging commands like `git restore -- .`, `git checkout -- .`, or `git add .`; only restore/stage exact files changed in the current turn after checking `git status --short`.
- When commit needed, group related changes that you just made. Related issues → one GitHub issue, don't split.
- User asks batched commits → split as small and focused as practical.
- Do not use type "any", eslint will not pass
- Always use the UI components from packages/ui, for example, button
- **Simplest solution always.** Don't add abstractions/helpers/wrappers unless clearly needed. Prefer deleting code over adding code. Fix only the bug, don't improve surrounding code.
- **Balance engineering depth.** Avoid over-engineering, over-debugging, and over-instrumentation, but do not use "simplicity" as an excuse for under-designed shortcuts that create future mess and technical debt. Choose the simplest design that still fits the expected product surface and failure modes.
- **不要 MVP 糊弄，做对的架构。** 不要默认提"先 A 凑合"的方案。先问"现在项目里有什么能复用的"。
- **Before evaluating refactor/migration value, ask WHY.** Forward-looking goals (reuse, new surfaces) reframe the answer.
- Do not use `pnpm -F @acme/db build` as a verification shortcut without permission; its `prebuild` runs migrations. Prefer `pnpm -F @acme/db typecheck` and focused lint/tests.
- Never run `lingui:extract` / `pnpm extract` / any i18n message-catalog regeneration unless the user explicitly asks. Don't decide on your own that "the strings changed so I should re-extract."

## Compression Rules

When compressing, preserve in priority order:

1. Architecture decisions (NEVER summarize)
2. Modified files and their key changes
3. Current verification status (pass/fail)
4. Open TODOs and rollback notes
5. Tool outputs (can delete, keep pass/fail only)

## Communication & Context Rules

- **All context goes to AGENTS.md, never to memory files** — ensures local dev and GitHub Actions agents see the same context.
- AGENTS.md is for durable project rules only. Do not add one-off task notes, self-reminders, temporary fixes, or implementation chatter.
- Use only Chinese and English
- Do not use agreement-as-apology or agreement-as-softener templates such as "You're right", "You're absolutely right", or "You're correct". State the facts, reason, and next step directly.
- 写作要连贯成完整判断，少用碎片化 bullet。除非用户明确要求清单，否则用短段落把因果、结论、下一步讲清楚；bullet 只用于步骤、选项、验收标准。

## Prompt & Agent Design Rules

- **Avoid overcorrection and prompt pollution.** If a schema/API/type contract already expresses a requirement, do not add extra prompt/tool-description rules to restate it. When a change is unnecessary or wrong, cleanly revert that change instead of adding more wording to explain around it.
- **Generic agent prompts must stay product-agnostic.** Growth prompts, skills, personas, cron prompts, and agent tool descriptions must not hardcode VibeCom's own positioning, ICP, communities, channels, examples, or defaults (technical founders, SaaS, indie hackers, MCP/IDE, etc.) unless the prompt is explicitly about VibeCom itself. User product positioning, audience, category, search surfaces, and examples must come from Product Context / channel context / runtime data; use cross-industry examples when examples are needed.
- **Prompt writing philosophy:** Give principles, not rigid rules. Tell the LLM what good output looks like and why — let it figure out how. Avoid prescribing exact fields/counts/formats unless output is machine-consumed.
- **System prompt** = WHO + cross-cutting rules. **Skill spec** = WHAT good output looks like. **LLM owns** strategy.
- Skill spec must NOT contain hardcoded queries, `save_document` calls, or principles already in system prompt.
- **Never remove subagent safety guards** — prompt restrictions, logging, and filtering in agent code are critical safety nets, not dead code.

## UI & Frontend Rules

- UI changes: responsive design, dark/light mode, and i18n (`<Trans>` / `` t` ``) are **not optional** — always include them.

## SEO & Link Design Rules

- Before changing marketing-page title, H1, metadata, FAQ, structured data, or major copy, check `docs/marketing/seo-keyword-map.md`. Keep one canonical primary keyword per page, preserve completed keyword intent, and update the map when targets change.
- External links opened from marketing pages, markdown, reports, chat references, showcase/source links, generated content, user-submitted content, or scraped references should default to `rel="nofollow noopener noreferrer"` when `target="_blank"` is used. Only omit `nofollow` for a deliberate editorial/partner link where passing authority is intended.
- Internal links should stay crawlable and should not use `nofollow`. Prefer normal `<Link>`/anchor paths with descriptive anchor text for SEO-critical navigation; avoid hiding important internal links behind JS-only interactions.
- Visible FAQ content and FAQ JSON-LD must stay one-to-one. If a question or answer changes, update both or remove the schema block.
- For new, renamed, or deleted marketing pages, check canonical metadata, sitemap registration, internal links, OG/social metadata, and any all-pages/index surfaces that list the page.
- Do not fabricate proof for SEO: verify external source URLs, social posts, claims, metrics, and testimonials before using them. Broken or unverifiable proof is worse than no proof.
- Images used as SEO/landing-page assets need descriptive alt text and stable dimensions; OG images should match the page's actual positioning and keyword intent.

## tRPC (Client-Side)

Use `useTRPC()` + `useQuery(trpc.*.*.queryOptions(...))` from @tanstack/react-query. Do NOT use `api.*.*.useQuery()`. Mutations: `trpc.*.*.useMutation()`.

## i18n (@lingui)

- JSX: `<Trans>Hello</Trans>` | Attributes: `` t`Search...` `` | Variables: define `msg`, render `_()`
- `<Trans>` and `t` only accept static strings, never wrap variables
- Never edit `.po` files — scripts auto-sync
- Imports: `msg` from `@lingui/core/macro`, `Trans/useLingui` from `@lingui/react/macro`

## Modals (NiceModal)

All modals use `@ebay/nice-modal-react`. One file per modal, register in `Modals.tsx`, open via `NiceModal.show()`. Modal owns its own data fetching. Reference: `DeleteChannelModal.tsx`.
